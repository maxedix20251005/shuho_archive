1 ER概要

エンティティ関係 必須

公開向け主要データ: `works`（作品）, `news_items`（お知らせ）

CVデータ: `enquiries`（問い合わせ）

管理・監査: `admin_users`（管理利用者）, `admin_audit_logs`（操作ログ）

関連: `admin_users.id` 1:N `works.created_by` / `works.updated_by`

関連: `admin_users.id` 1:N `admin_audit_logs.actor_admin_id`

注意: 体験申込専用テーブルは作成しない（現スコープ外）

2 テーブル定義

共通方針 必須

・DBはSupabase PostgreSQLを使用

・主キーは `uuid`（`gen_random_uuid()`）を採用

・日時は `timestamptz`（UTC保存）を採用

・論理削除が必要なテーブルは `deleted_at` を使用

`admin_users` / `admin_audit_logs` 必須

・`admin_users`: Supabase Auth `auth.users` と1:1で運用（`id(uuid)` を共通）

・`admin_users.role`: `admin` / `editor`

・`admin_audit_logs`: `actor_admin_id`, `action`, `target_table`, `target_id`, `meta_json`, `created_at` を保持

・監査ログは削除不可（append-only）で運用する

RLS設計（Supabase） 必須

・`works`: 公開サイトは `publish_status='published' and deleted_at is null` のみ `SELECT` 許可

・`news_items`: 公開サイトは `publish_status='published'` のみ `SELECT` 許可

・`enquiries`: 現行プロトタイプは匿名 `INSERT/SELECT/UPDATE` を許可（静的管理画面用）。本番前に管理ロール専用へ切替必須

・`admin_users` / `admin_audit_logs`: 管理ロールのみアクセス可

・Service Role Keyはサーバーサイド限定、静的HTML側には露出させない

4 初期データ / 運用データ

初期投入データ（推奨） 必須

・`works`: 各カテゴリ最低3件（合計15件以上）を投入して絞り込み挙動を確認

・`news_items`: 公開状態のダミー3件（`YYYY.MM.DD` 表示形式確認用）

・`admin_users`: `admin` 1名、`editor` 1名

マスタ / 列挙値管理 必須

・カテゴリ、公開状態、問い合わせ状態はDB CHECK制約で統制する

・将来的に項目追加が頻発する場合のみ、別途マスタテーブル化を検討する

補足（実装方針） 必須

・静的HTMLからの動的データ取得は Supabase JS で公開APIを参照する

・問い合わせ送信はAPI経由（Edge Functionまたは安全なサーバー処理）を推奨する

・画像は Supabase Storage の専用バケット（例: `works-public`）で管理する
